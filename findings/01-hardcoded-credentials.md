# InsecureShop - Hardcoded Credentials

## 1. 개요

`InsecureShop`의 로그인 기능을 분석한 결과, 사용자 인증에 사용되는 계정 정보가 애플리케이션 내부 코드에 하드코딩되어 있음을 확인하였다. 공격자는 APK를 디컴파일하여 자격증명을 식별할 수 있으며, 별도의 서버 검증 우회 없이도 해당 값을 이용해 정상 로그인에 성공할 수 있다.

로그인 실패 동작을 먼저 확인한 뒤, `jadx`를 이용해 인증 흐름을 추적하고, 하드코딩된 자격증명을 식별한 후 실제 로그인 성공 여부까지 검증하였다.

## 2. 취약점 요약

| 항목 | 내용 |
|---|---|
| 취약점명 | `Hardcoded Credentials` |
| 취약점 유형 | 클라이언트 내 민감정보 하드코딩 |
| 영향 | 디컴파일만으로 자격증명 추출 가능, 정상 사용자 권한 획득 가능 |
| 분석 도구 | `jadx`, `adb`, `Nox` |
| 난이도 | 낮음 |

## 3. 분석 환경

| 항목 | 내용 |
|---|---|
| 대상 앱 | `InsecureShop` |
| 실행 환경 | `Nox` |
| 운영체제 | Android |
| 정적 분석 | `jadx` |
| 동적 검증 | `adb` |

## 4. 분석 방법

분석은 다음 순서로 진행하였다.

1. 로그인 화면에서 임의의 값을 입력하여 기본 동작을 확인하였다.
2. `jadx`에서 `LoginActivity`의 로그인 처리 메서드를 확인하였다.
3. 입력값이 전달되는 인증 함수와 실제 비교 로직을 추적하였다.
4. 하드코딩된 자격증명을 식별한 뒤 실제 로그인에 사용해 성공 여부를 검증하였다.

## 5. 상세 분석

### 5.1 로그인 실패 동작 확인

우선 로그인 화면에서 임의의 아이디와 비밀번호를 입력한 뒤 `Log in` 버튼을 선택하였다. 그 결과 `Invalid username and password` 메시지가 출력되었으며, 입력값이 인증 로직을 거쳐 실패 분기로 처리되는 것을 확인하였다.

### 5.2 로그인 처리 함수 추적

`jadx`에서 `LoginActivity`를 확인한 결과, `onLogin()` 메서드에서 사용자가 입력한 `username`, `password`를 가져온 뒤 `Util.verifyUserNamePassword(username, password)`로 전달하는 흐름을 확인하였다.

실제 인증 여부는 `LoginActivity` 내부가 아니라 `Util` 클래스의 검증 함수에서 결정되고 있었다. 따라서 취약점 분석의 핵심은 `verifyUserNamePassword()`와 그 하위 로직을 따라가는 것이다.

### 5.3 하드코딩된 자격증명 식별

인증 함수를 추적한 결과, `verifyUserNamePassword()`는 내부적으로 `getUserCreds()`를 호출하여 계정 정보를 조회하고 있었다. `getUserCreds()` 메서드에서는 `HashMap<String, String>` 객체를 생성한 뒤, 아래와 같이 사용자 계정 정보를 직접 코드에 선언하고 있었다.

```java
userCreds.put("shopuser", "!ns3csh0p");
```

이 값은 외부 입력이나 서버 응답으로 받아오는 정보가 아니라 애플리케이션 내부에 고정된 값이다. 따라서 APK 파일만 확보하면 누구나 디컴파일을 통해 계정 정보에 접근할 수 있다.

### 5.4 로그인 성공 검증

식별한 자격증명 `shopuser / !ns3csh0p`를 실제 로그인 화면에 입력한 결과, 앱은 `ProductListActivity`로 이동하며 로그인 성공 상태를 보여주었다.

이를 통해 본 취약점은 단순히 "코드 안에 문자열이 존재한다" 수준이 아니라, 실제 인증 절차를 우회 가능한 형태로 악용될 수 있음을 확인하였다.

## 6. 영향도

하드코딩된 자격증명은 클라이언트 애플리케이션 내부에 포함되므로, 공격자가 APK를 확보하는 것만으로도 손쉽게 추출할 수 있다. 실제 서비스 환경에서 이와 같은 구조가 존재할 경우 다음과 같은 문제가 발생할 수 있다.

- 공격자가 정상 사용자 권한으로 로그인할 수 있다.
- 인증 구조가 서버가 아닌 클라이언트에 의존하게 되어 접근 제어가 무력화된다.
- 앱이 배포되는 순간 동일한 자격증명이 다수 사용자에게 노출될 수 있다.

즉, 이 취약점은 단순한 구현 실수가 아니라 인증 체계 전반의 신뢰 경계를 잘못 설정한 문제로 볼 수 있다.

## 7. 대응 방안

- 자격증명, API 키, 비밀값 등 민감 정보는 애플리케이션 코드에 하드코딩하지 않아야 한다.
- 사용자 인증은 서버 측에서 수행하고, 클라이언트는 검증 결과만 전달받도록 설계해야 한다.
- 클라이언트 내부 값만으로 접근 권한이 결정되는 구조는 지양해야 한다.
- 난독화는 분석 난도를 높일 수는 있지만, 하드코딩 문제를 근본적으로 해결하지는 못한다.

## 8. 결론

이번 분석에서는 `jadx`를 이용해 로그인 처리 흐름을 추적하고, `getUserCreds()` 메서드에 하드코딩된 자격증명을 식별한 뒤, 실제 로그인 성공까지 검증하였다. 이를 통해 Android 애플리케이션에서 하드코딩된 인증정보가 얼마나 쉽게 식별되고 악용될 수 있는지 확인할 수 있었다.

이 사례는 정적 분석만으로도 인증 우회 가능성을 입증할 수 있는 대표적인 예시다.
